# v1.1 Update System Design

## Overview

Add automatic update checking and in-app update installation to Paper Suitecase using the Sparkle 2 framework. Includes branching workflow changes for structured release management.

## Branching & Release Workflow

### Branch structure
- `main` — release-only, every commit is a tagged release
- `dev` — working branch, all PRs merge here
- Feature branches (`feature/xxx`) branch from `dev`

### Release flow
1. Work accumulates on `dev` via PR merges
2. Bump version in `pubspec.yaml` on `dev`
3. Merge `dev` → `main`
4. Tag `main` with version (e.g. `v1.0.1`)
5. Build macOS app, zip, sign with EdDSA
6. Create GitHub Release with zip attached
7. Update `appcast.xml` in website repo with new entry

### Version numbering
- `v1.0.x` — patches (bug fixes, small improvements) leading up to v1.1
- `v1.1.0` — the update system itself ships
- `pubspec.yaml` format: `version: X.Y.Z+N` where N increments with each build

## Sparkle 2 Integration Architecture

### How it works
Sparkle 2 is a native macOS framework integrated into the Xcode runner (not the Flutter/Dart layer). It checks an appcast XML file for new versions and handles download, EdDSA signature verification, app bundle replacement, and relaunch.

The app is **not sandboxed** (uses direct filesystem access for entries). Sparkle 2 in non-sandboxed mode handles installation directly without XPC services.

### Components

**1. Appcast file (`appcast.xml`)**
- Hosted at `https://initialneil.github.io/papersuitecase/appcast.xml`
- Contains version entries with: download URL (GitHub Releases), file size, EdDSA signature, `sparkle:minimumSystemVersion` (set to `12.0` matching the app's deployment target)

Example `<item>` template:
```xml
<item>
  <title>Version 1.1.0</title>
  <pubDate>Sun, 23 Mar 2026 00:00:00 +0000</pubDate>
  <sparkle:version>1.1.0</sparkle:version>
  <sparkle:shortVersionString>1.1.0</sparkle:shortVersionString>
  <sparkle:minimumSystemVersion>12.0</sparkle:minimumSystemVersion>
  <enclosure
    url="https://github.com/initialneil/papersuitecase-app/releases/download/v1.1.0/PaperSuitecase-macOS-v1.1.0.zip"
    length="25000000"
    type="application/octet-stream"
    sparkle:edSignature="SIGNATURE_HERE"
  />
</item>
```

**2. EdDSA key pair**
- Generated once with Sparkle's `generate_keys` tool
- Private key stored in macOS Keychain (never committed to git)
- Public key embedded in `macos/Runner/Info.plist`

**3. macOS native integration**
- Sparkle 2 framework added via CocoaPods (`macos/Podfile`): `pod 'Sparkle', '~> 2.0'`
- Configured in `Info.plist`:
  - `SUFeedURL` → appcast URL
  - `SUPublicEDKey` → EdDSA public key
  - `SUScheduledCheckInterval` → `86400` (24 hours)
  - `SUAutomaticallyUpdate` → `NO` (user controls when to install)
  - `SUEnableAutomaticChecks` → `YES` (skip first-launch prompt, always check)

**4. Flutter ↔ native bridge (`MethodChannel`)**

This is the first MethodChannel in the project. Channel name: `com.papersuitecase/sparkle`

**Two Sparkle APIs used for different purposes:**
- **Silent background check**: `SPUUpdater` with custom `SPUUpdaterDelegate` implementing `updater(_:didFindValidUpdate:)` and `updaterDidNotFindUpdate(_:)`. Reports status to Dart without showing any native UI.
- **User-initiated update dialog**: `SPUStandardUpdaterController.checkForUpdates(_:)`. Opens Sparkle's native dialog showing release notes, download progress, install & relaunch. Used for both "Check for Updates" and "Install Update" buttons (same operation — Sparkle decides whether to show "up to date" or the install dialog based on current state).

Dart → native methods:
- `checkForUpdates()` — triggers `SPUStandardUpdaterController.checkForUpdates(_:)`, opens native Sparkle dialog
- `getUpdateStatus()` — returns `{available: bool, version: String?}` from cached state

Native → Dart callbacks (via `invokeMethod` on the channel):
- `onUpdateAvailable(version)` — called when `SPUUpdaterDelegate.updater(_:didFindValidUpdate:)` fires
- `onUpdateCheckError(error)` — called when check fails (network error, malformed appcast, etc.)

**Native side registration pattern** (first MethodChannel in the project):
```swift
// In AppDelegate.swift applicationDidFinishLaunching:
let controller = mainFlutterWindow?.contentViewController as! FlutterViewController
let channel = FlutterMethodChannel(
    name: "com.papersuitecase/sparkle",
    binaryMessenger: controller.engine.binaryMessenger
)
let sparklePlugin = SparklePlugin(channel: channel)
channel.setMethodCallHandler(sparklePlugin.handle)
```

**5. Signing releases**
- After building zip: `sign_update PaperSuitecase-macOS-vX.Y.Z.zip`
- Output EdDSA signature added to appcast `<item>` `sparkle:edSignature` attribute

## UI Changes

### Settings icon badge (`tag_sidebar.dart`)
- Wrap existing settings `IconButton` in a `Stack`
- Add positioned 8px teal dot (`Color(0xFF4EB8A1)`) in top-right corner
- Dot visible only when `appState.showUpdateBadge == true`
- Dot clears when user opens settings (`markUpdateBadgeSeen()`)

### Settings page — About section (`settings_view.dart`)
New section at the bottom of the settings ListView:

- **Current version**: `v1.1.0` (read via `package_info_plus` — new dependency)
- **Update status line** (three states):
  - "Checking..." (gray, shown while initial check is in progress)
  - "Up to date" (gray) when no update available
  - "Version X.Y.Z available" (teal) when update found
  - "Could not check for updates" (red) on error, with retry affordance
- **"Check for Updates" button**: always visible, calls `checkForUpdates()` via MethodChannel — opens Sparkle's native dialog
- When update is available, the status line itself is tappable and also triggers the Sparkle dialog

Uses existing `_SectionHeader` and `_SettingBox` patterns from the current settings view.

### AppState additions (`app_state.dart`)
```
bool _updateAvailable = false
String? _latestVersion = null
String? _updateCheckError = null
bool _hasSeenUpdateBadge = false
String _currentVersion = ''  // loaded from package_info_plus at init

bool get updateAvailable
String? get latestVersion
String? get updateCheckError
String get currentVersion
bool get showUpdateBadge  // updateAvailable && !hasSeenUpdateBadge

void markUpdateBadgeSeen()  // called when settings opened
Future<void> checkForUpdates()  // invokes MethodChannel, opens Sparkle dialog
void _onUpdateAvailable(String version)  // callback from native
  // - compares version against _latestVersion to deduplicate
  // - resets _hasSeenUpdateBadge if version changed (new update for a different version)
  // - persists to SharedPreferences for badge survival across restarts
void _onUpdateCheckError(String error)  // sets _updateCheckError, notifies
```

**Persistence:** `_updateAvailable` and `_latestVersion` are cached in `SharedPreferences` so the badge survives app restarts. Cleared when the app version matches or exceeds the cached latest version.

### Native side files
- `macos/Runner/SparklePlugin.swift` — MethodChannel handler, wraps both `SPUUpdater` (silent checks) and `SPUStandardUpdaterController` (user-facing dialog)
- `macos/Runner/AppDelegate.swift` — register SparklePlugin channel using FlutterViewController binaryMessenger pattern

### New dependency
- `package_info_plus` in `pubspec.yaml` — for reading current app version at runtime

## Update check flow

1. App launches → `AppDelegate` initializes `SPUUpdater` (silent) and `SPUStandardUpdaterController` (for user-triggered dialog)
2. `AppState.initialize()` loads cached update status from `SharedPreferences`, loads current version from `package_info_plus`
3. If cached `latestVersion` > `currentVersion`, immediately set `updateAvailable = true` (badge shows without waiting for network)
4. Sparkle's `SPUUpdater` automatically checks appcast (on launch + every 24h per `SUScheduledCheckInterval`)
5. `SPUUpdaterDelegate.updater(_:didFindValidUpdate:)` fires → native sends `onUpdateAvailable(version)` to Dart via MethodChannel
6. `AppState._onUpdateAvailable()` compares version, updates state, persists to SharedPreferences, resets badge if version changed → teal dot appears on settings gear
7. User opens settings → `markUpdateBadgeSeen()` clears dot, About section shows version + status
8. User taps "Check for Updates" → calls native `checkForUpdates()` → Sparkle opens its standard native dialog (shows "up to date" or update with release notes, download, install, relaunch)
9. If check fails (network, malformed appcast) → `_onUpdateCheckError()` sets error state → settings shows "Could not check for updates"

## Release build pipeline

### One-time setup
1. `generate_keys` → creates EdDSA key pair, stores private key in Keychain
2. Add public key to `macos/Runner/Info.plist` as `SUPublicEDKey`
3. Add `pod 'Sparkle', '~> 2.0'` to `macos/Podfile`, run `pod install`
4. Add `package_info_plus` to `pubspec.yaml`
5. Create initial `appcast.xml` in website repo (`initialneil/papersuitecase`)

### Per-release steps (manual)
1. Bump version in `pubspec.yaml` (version + build number)
2. Merge `dev` → `main`, push
3. Tag: `git tag vX.Y.Z && git push --tags`
4. Build: `flutter build macos --release`
5. Zip: `ditto -c -k --keepParent "paper_suitecase.app" PaperSuitecase-macOS-vX.Y.Z.zip`
6. Sign: `sign_update PaperSuitecase-macOS-vX.Y.Z.zip` → copy EdDSA signature
7. Update `appcast.xml`: add new `<item>` using template above (fill version, URL, signature, file length)
8. Create release: `gh release create vX.Y.Z PaperSuitecase-macOS-vX.Y.Z.zip --title "..." --notes "..."`
9. Push appcast update to website repo

Automation via GitHub Actions is out of scope for v1.1 but could be added later.

## Out of scope
- GitHub Actions CI/CD for automated builds
- Windows/Linux support
- Delta updates (Sparkle supports these but full zip replacement is fine to start)
- Apple code signing / notarization (can be added independently)
- Release notes display within the Flutter app (Sparkle's native dialog shows these)
- App sandboxing (app is unsandboxed, Sparkle handles installation directly)
